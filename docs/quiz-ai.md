# 早押しクイズAI — Claude Code Handoff

## プロジェクト概要

競技早押しクイズに特化した2モデル構成のAIシステム。
問題文を音声でリアルタイム受信し、人間と同条件（ハンデなし）で早押し対戦できるレベルを目指す。

先行研究として東北大NLPグループの JAQKET / RCQA コーパス、AI王コンペ（東北大・理研・NTT・baton・キオクシア）が存在する。
先行研究との差別化軸：**Reasoningの獲得**、**ゲーム理論的戦略**、**クイズ文法の学習**、**GRPO強化学習**、**音声入力**。

### 2モデル構成

```
┌─────────────────────────────────────────────┐
│              割り込みモデル                    │
│  LiquidAI/LFM2.5-350M（hybrid conv+attn）    │
│  役割：毎文字ごとにbuzz/no-buzzを閾値判定      │
│  目標速度：700〜1200 t/s                      │
│  Reasoning採用（短いthink→confidence出力）    │
│  精度/量子化/回帰ヘッド化はPhase 0で確定       │
└─────────────────────────────────────────────┘

┌─────────────────────────────────────────────┐
│              メインLLM                        │
│  Qwen3.5-9B-Base（INT8）または               │
│  Gemma-4-E4B（選定中、後述）                 │
│  役割：部分的な問題文から回答を予測            │
│  特徴：投機的推論、ゲーム戦略認識              │
└─────────────────────────────────────────────┘
```

### パイプライン（確定）

```
音声入力
  ↓
[Silero VAD]  無音区間の検出
  ↓
[parakeet-tdt_ctc-0.6b-ja]  Streaming STT
  出力：文字ストリーム。最末尾の未確定単語のみひらがな出力。
  例：「743年に聖武てんの…」→「墾田永年私財法」
  ※ CTCのフレームアライメントを使い未確定セグメントを分離する
  ↓ 文字が来るたびに
[割り込みモデル]  buzz/no-buzz閾値判定（現在の文字列末尾を前提に毎回推論）
  ↑ 同時並行
[メインLLM]  投機的Reasoning開始（buzzが確定する前から推論開始）
  - no-buzzと判定された瞬間：推論を即座に打ち切り、KVキャッシュ破棄
  - buzz確定：そのまま推論を継続（生成を止めない）
  - buzz位置と投機前提が不一致：推論をリセット
  ↓ buzz確定後
ボタン押しシグナル送信（実装検討中）
  ↓
[メインLLM]  回答テキスト生成完了
  ↓
[Irodori-TTS-500M-v3]  音声合成
  ↓
無音削除（軽量処理）
  ↓
音声出力・再生
```

## 主要な設計判断（確定済み）

### 割り込みモデルの判定方式
閾値ベース。ゲーム状態と連動した動的閾値も検討：

```python
# 基本形
if buzz_score > θ:
    trigger_buzz()

# 動的閾値（ゲーム状態と連動）
θ_dynamic = base_θ * game_state_factor
```

**ゲーム状態は割り込みモデルには渡さない（確定）。** 割り込みモデルは純粋な buzz_score 出力に専念し、速度を最優先する。動的閾値 `θ_dynamic = base_θ × game_state_factor` はオーケストレータ側で外付け調整する。ゲーム戦略はメインLLM ＋ 外付け閾値で担保。

### ゲーム戦略：システムプロンプト動的注入（確定）
専用戦略LLMは使わない。メインLLMのシステムプロンプトにゲーム状態を毎問注入する。
相手プレイヤーの回答速度傾向の分析もReasoningに含める。

```
[GAME_STATE]
自分: 7点, 相手: 5点, 残り問題: 8問
相手の平均buzz位置: 問題文の42%地点 / 最近3問: 連続正解
推奨戦略: やや慎重、後半狙い
[/GAME_STATE]
問題文（途中まで）: {partial_question}
```

### 投機的推論とKVキャッシュ（確定）
- buzz確定前から推論開始
- no-buzz時はKVキャッシュを捨てる（保持よりもバッチスループット優先）
- buzz時は生成継続（ストップなし）
- buzz位置の前提ズレはリセット
- 投機推論は常に「現在の文字列末尾」を前提とする

### 対戦相手（確定）
`openai/gpt-oss-120b:cerebras` (HuggingFace Inference, 実測1,145 t/s)
Vast.ai RTX5090インスタンスとHF Inference Providersを組み合わせる。

### クイズ文法の学習（採用確定）
競技クイズ問題文の構造を学習させる：

```
[導入・フェイント] → [絞り込み情報] → [決定的情報] → 答え
```

- パラレル（「〜ですが〜」構文）
- 読点押し（「〜、〜」の読点でbuzz）
- などの定型パターンを訓練データに組み込む

## 実装フェーズ

### Phase 0：F/S検証（既存モデルのみ）★現在のフォーカス（2026-05-30〜）
**目的：** パイプライン疎通・レイテンシ計測・ボトルネック特定
**方針：** コーパス合成（Phase 1準備）は一時停止し、先にパイプラインの妥当性を検証する。
段階化：
- **Phase 0a（テキストのみ・GPU不要）**：char stream → asyncio → ルールベースbuzz(65%) → gpt-oss-120b投機推論（buzz前開始/no-buzz打ち切り）。メインLLMのprefix正解率＋LLM経路レイテンシを計測。パイロットの100問アノテ結果を再利用。
- **Phase 0b（音声・Vast.ai 5090）**：parakeet STT + Silero VAD + Irodori TTS を統合しE2Eレイテンシ計測。

- parakeet Streaming STT の動作確認
- 割り込みモデルは**ルールベース仮実装**（問題文の**65%地点**で強制buzz開始。50/65/80%でA/B比較し誤答率とのトレードオフを見る）
- メインLLMは `gpt-oss-120b:cerebras` で代替
- TTS・VAD の統合テスト
- エンドツーエンドレイテンシの計測
- LFM2.5-350Mの実機速度検証（フル精度/量子化/回帰ヘッドの要否判定）

**成功基準（確定・たたき台）：**
- パイプラインがクラッシュせず1問完走（STT→割り込み→メイン→TTS）
- 全文入力での正解率 **≥ 60%**（下回ったら正規化/プロンプトを疑う）
- **buzz確定 → 音声出力開始 ≤ 1.0s**（割り込み判定自体は新規文字到着から ≤ 300ms）

### Phase 1：SFT
**目的：** 両モデルに早押しクイズドメインを教え込む

#### 割り込みモデルのSFT
- 入力：部分問題文（N文字目まで）
- 出力：`<think>{短い推論}</think>{confidence:.2f}`（corpus-1形式・Reasoning採用）
- 訓練データ：corpus-1（S-buzzアノテーション由来）
- 正解buzz位置の定義：**S-buzz自動アノテーション（確定）** — corpus.md Step 1（binary search, k=5, threshold 0.8）
- SFT loss：**CLM（テキスト生成）で開始**。Phase 0の速度検証で不足なら回帰ヘッド（最終トークン hidden→linear→sigmoid、BCE/MSE）にフォールバック

#### メインLLMのSFT
- 入力：部分問題文 + ゲーム状態プロンプト
- 出力フォーマット：**案B `<think>{reasoning}</think>{answer}`（確定）**。確信度は割り込みモデルが担うため案Cは不採用
- **think budget + adaptive thinking（確定）**：難易度（≒buzz_ratio）で reasoning長を変える。簡単=短い/no-think、難問=長いthink。一部を short-think/no-think 変種としても生成し思考量変調を学習させる
- 投機推論中のCoT打ち切り：`</think>` 検出で answer フェーズへ。推論時は max think tokens で上限を切り強制 `</think>`
- クイズ文法フェーズのアノテーションを付与するか **未決定**

#### 共通SFT設定（未決定）
- フレームワーク：Unsloth / TRL
- LoRA rank / alpha：両モデルとも未定
- max_seq_length：問題文の最大長に合わせて設定（典型的な早押しクイズ問題文の長さを事前調査）

#### SFT→GRPO移行判断基準（確定）
- 割り込みモデル：buzz位置 **MAE ≤ 8文字**（val）
- メインLLM：corpus-2 val の **answer正解率 ≥ 65%**
- 両方クリアでGRPOへ

### Phase 2：GRPO
**目的：** ゲームの中で勝つための戦略を強化学習で獲得させる

#### エピソード定義（確定・2段階）
```
GRPO-1（warmup）: 案A 1問1答 — buzzタイミングと正解率を安定させる
GRPO-2（戦略）  : 案B N問連続（N=7前後）— ゲーム状態に意味が出て戦略を獲得
案C（先取り）    : 評価用の対戦フォーマットに使う
```

#### 報酬関数（確定）

両モデルに**同一スカラーを共同責任で返す**（システム全体最適のため）。

```
R(共通・両モデルに同値):
  誤答（buzz後に外す）          : -1.5
  正解                          : 1.0 + 0.5 × (1 - buzz/length)   # 1.0〜1.5、早いほど高
  相手に先取りされた(機会損失)   : -0.5
  答えられたのにスルー(両者×)    : -0.1
```

- **バランスのガードレール：`|機会損失| < |誤答|` を必ず維持**（-0.5 < -1.5）。機会損失ペナルティを誤答に近づけると「先取りされるくらいなら当てずっぽうでbuzz」する無謀戦略を学習する。重くしたい場合でも -0.6 が上限目安。
- 共同責任のため buzz誤答は割り込み・メイン双方に同符号で配る。信用割当ノイズで学習が不安定なら、共通報酬をベースに各モデル固有の微小shaping項を足す（デフォルトは共通のみ）。

#### GRPO設定（未決定）

| 項目 | 内容 |
|---|---|
| グループサイズG | 未定 |
| KL係数β | 両モデルで別々に設定するか |
| 2モデルの更新タイミング | 同時か交互か |
| rollout生成 | vLLM使用有無 |
| リファレンスポリシー | 初期固定か定期更新か |

## Claude Code への作業依頼時の注意事項

- 推論時、STT/割り込みモデル/メインLLM/TTS は**全て単一のVast.ai RTX 5090 32GB**上で動作する（訓練はModal A100、本節「ハードウェア・インフラ」参照）
- parakeet は **CTC** アーキテクチャのため、トークンを文字単位でストリーミング出力できる（Attentionモデルと異なり全発話完了を待たない）
- 割り込みモデルは **LiquidAI/LFM2.5-350M**（hybrid conv+attn、オンデバイス高速推論向け）。毎文字判定の速度クリティカル経路
- メインLLMの投機推論と割り込みモデルの推論は**非同期**で並行動作する
- TTS後に次の問題が始まるため、TTS生成中の先読みSTTは**不要**
- Modalはサーバーレスのため**ライブ推論には使わない**（訓練・バッチ評価専用）。Vast.aiは起動・停止コストを考慮し長時間セッション前提
